# AGENTS.md

## Highest priority: Andrej Karpathy rules

Эти правила имеют наивысший приоритет и должны применяться раньше остальных инструкций в этом файле.

### 1. Think before coding

- Не скрывать путаницу и не делать поверхностные компромиссы
- Перед внедрением явно формулировать предположения
- Если есть неуверенность, задавать вопрос
- Если есть несколько интерпретаций задачи, перечислять их, а не выбирать молча
- Если есть более простой подход, прямо говорить об этом
- Если что-то неясно, остановиться, назвать источник неясности и запросить уточнение

### 2. Simplicity first

- Писать минимальный код, который решает проблему
- Не добавлять функции за пределами запроса
- Не вводить абстракции для одноразового кода
- Не добавлять гибкость или конфигурируемость без явного запроса
- Не писать обработку ошибок для невозможных сценариев
- Если решение можно выразить заметно меньшим объёмом кода, упрощать его
- Если старший инженер счёл бы решение слишком сложным, его нужно упростить

### 3. Surgical changes

- Менять только то, что необходимо для выполнения запроса
- Убирать только тот беспорядок, который создают собственные изменения
- Не улучшать соседний код, комментарии или форматирование без необходимости
- Не рефакторить то, что не сломано
- Следовать существующему стилю проекта
- Если замечен несвязанный мёртвый код, упомянуть его, но не удалять без запроса
- Удалять только те импорты, переменные и функции, которые стали неиспользуемыми из-за текущих изменений
- Не удалять ранее существовавший мёртвый код без явной просьбы
- Каждая изменённая строка должна напрямую отслеживаться до запроса пользователя

### 4. Purposeful execution

- Сначала определять проверяемые критерии успеха
- Работать циклом до верификации результата
- Превращать задачи в проверяемые цели
- Для исправления бага сначала по возможности воспроизводить проблему тестом, затем добиваться прохождения
- Для добавления проверки сначала писать тесты на невалидные входы, затем добиваться прохождения
- Для рефакторинга убеждаться, что тесты проходят до и после изменений
- Для многошаговых задач фиксировать краткий план в формате `Step -> verify`
- Сильные критерии успеха важнее расплывчатого "сделать, чтобы работало"

## General operational rules

- Рассматривать репозиторий как production-grade проект: действовать бережно к коду, данным, конфигам и пользовательским сценариям
- Перед любым заметным изменением сначала убедиться, что задача понята, риски названы, а неоднозначности при необходимости уточнены
- После каждого нового задания кратко формулировать, что именно понято и как будет трактоваться задача
- Для простых локальных и низкорисковых правок, если запрос однозначен, можно сразу переходить к реализации без отдельного подтверждения
- Любые изменения, влияющие на рантайм, конфиги, окружение, контейнеры, entrypoint, миграции или серверные файлы, сначала должны быть оформлены в репозитории
- Не допускать состояния, в котором фактический рабочий код или конфигурация расходятся с репозиторием, кроме явно согласованного временного исключения
- Перед завершением задачи обязательно проверять, что решение рабочее
- Минимум верификации: запускать релевантные проверки вроде тестов, линтера, сборки или валидации изменённых файлов, если это можно сделать локально
- Не завершать задачу с непроверенными изменениями, если локальная проверка доступна
- Если задача затрагивает протоколы, обязательно учитывать и проверять поведение для обоих поддерживаемых в проекте протоколов, а не только для одного из них
- Не делать commit автоматически без явной команды пользователя
- Команда `push` или `пуш` считается явным разрешением и на commit, и на push, если пользователь не уточнил иное
- Если push невозможен из-за доступа, сети или конфликта, нужно явно сообщить причину и предложить следующий шаг
- Отвечать пользователю коротко и по делу, если он не просил подробности
- Не расширять объём простых точечных задач без прямого запроса

## Default stack

Использовать по умолчанию:

- Kotlin
- Jetpack Compose
- Material 3
- Navigation Compose
- MVVM + Repository
- ViewModel + StateFlow
- Kotlin Coroutines + Flow
- Retrofit + OkHttp
- kotlinx.serialization
- Hilt
- Room только при реальной необходимости локальной структурированной БД, кеша или offline-first
- DataStore для настроек и простых локальных значений
- Gradle Kotlin DSL
- Version Catalog
- KSP, если библиотека поддерживает

Не использовать без явной причины:

- Java
- XML layouts
- Fragments
- LiveData
- RxJava
- Raw Dagger
- Koin
- Ktor Client
- Firebase как дефолтное решение
- WebView вместо native UI
- Multi-module архитектуру на старте
- Чрезмерную Clean Architecture для простых CRUD-фич

## Architecture

Придерживаться простой layered architecture:

`UI -> ViewModel -> Repository -> API / DB / DataStore`

Правила:

- UI не должен знать про сеть, базу, DTO, Entity и бизнес-логику
- ViewModel управляет состоянием экрана и действиями пользователя
- Repository скрывает источники данных
- Domain layer и UseCase-классы добавлять только при реальной необходимости
- DTO, Entity, Domain model и UI model не смешивать
- Использовать mapper-слой там, где это оправдано

## Compose rules

- По возможности делать экраны stateless
- Route/container composable может работать с ViewModel
- Content composable должен получать state и callbacks
- Не выполнять сеть, БД, DataStore или бизнес-логику внутри composable
- Состояние экрана держать через `UiState`
- Действия пользователя оформлять через `UiAction` или аналогичный явный контракт
- Одноразовые события отделять от постоянного состояния экрана

## Network and API

- Для REST использовать Retrofit + OkHttp + kotlinx.serialization
- Не хардкодить `baseUrl`, токены и чувствительные значения
- Ошибки API обрабатывать единообразно
- DTO не должны протекать в UI
- Для потенциально растущих списков предусматривать pagination
- Не логировать токены, пароли и персональные данные

## Backend defaults

Если в рамках задачи требуется backend, предполагаемый стек:

- Python
- FastAPI
- PostgreSQL
- SQLAlchemy 2.x
- Alembic
- Pydantic
- pytest
- OpenAPI как контракт между backend и Android

Требования к API:

- версионирование через `/api/v1`
- стабильные response-схемы
- явные error codes
- понятный auth flow
- ISO 8601 UTC для дат
- cursor-based pagination для больших списков
- не возвращать ORM-модели напрямую

## Local storage

- DataStore использовать для настроек, флагов и простых значений
- Room использовать только для структурированных данных, кеша, offline-first или сложных локальных запросов
- Не использовать Room вместо настроек
- Не использовать DataStore вместо базы данных
- Не применять destructive migration в production без явного согласования

## Security

- Использовать только HTTPS
- Не хранить секреты в коде или APK
- Не логировать чувствительные данные
- Минимизировать permissions
- При `401` не делать бесконечные retry
- Logout должен очищать локальное состояние пользователя
- Если нужно хранить токены или другие чувствительные данные, сначала предложить безопасную стратегию и объяснить trade-offs

## Scaling

- Не усложнять проект заранее
- Multi-module, отдельный domain layer, use cases, offline-first, сложный кеш, background workers и прочие усложнения добавлять только при реальной причине
- Если предлагается усложнение, нужно кратко объяснить:
  - зачем оно нужно
  - что сейчас проще
  - что потом лучше масштабируется
  - можно ли это отложить

## Code quality

- Писать production-ready код
- Не использовать deprecated API
- Не добавлять зависимости без необходимости
- Не смешивать ответственность слоев
- Не создавать god-классы
- Не оставлять псевдокод вместо реализации
- Не создавать абстракции на всякий случай
- При изменении архитектурного решения явно объяснять причину
